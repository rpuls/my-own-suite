My Own Suite can run in different places depending on how hands-on you want to be.

You do not need to think about every infrastructure detail first. Start by choosing between two main paths.

## Which option fits you?

### Platform

You want the simplest setup and do not mind using a hosted platform that manages most infrastructure details for you.

Recommended choice: [Deploy on Railway](/docs/deploy-on-railway)

Why people choose this:

- Very little manual setup
- Fastest path to a working suite
- No IT skills required
- Best fit for non-technical users or anyone who wants to try the project quickly

Tradeoff:

- It runs on a managed platform operated by a company, so pricing and platform rules are outside your control.
- Platform deployments may not expose every self-host maintenance feature because updates, backups, and infrastructure controls usually belong to the hosting platform.

### Self-hosted

You want My Own Suite to run on infrastructure you control, with your installed suite managing routing, updates, backups, restore, and customization as one self-hosted system.

Recommended choices:

- [Self-host on a cloud server](/docs/deploy-on-vps)
- [Self-host on your own hardware](/docs/deploy-on-your-own-hardware)

Why people choose this:

- You can choose your server provider
- You can run the same self-hosted setup on a cloud machine or your own box
- You get the path where in-suite update, backup, restore, HTTPS, and customization work matters most
- Potentially low recurring cost if you already own suitable hardware
- A strong fit if you want hosting freedom or full local ownership

Tradeoff:

- The supported self-hosted setup targets `Ubuntu Server 24.04 LTS`, not arbitrary Linux distributions.
- Cloud-server setup still needs a server provider and DNS.
- Own-hardware setup still needs local networking decisions, especially DNS.

## Access style

The biggest practical difference between the two self-hosted installers is where the suite naturally lives.

Cloud-server self-hosting is the easiest fit when you want secure browser access from anywhere. The server already lives on the internet, and some providers may even give you a temporary HTTPS address while you connect your own domain.

Own-hardware self-hosting starts from a more private posture. The suite can live quietly on your home or office network, and you decide if or when to add remote access later. That makes it the strongest fit when local control matters most, while still leaving a path to outside access when you are ready.

## Self-hosted installers

The self-hosted path has two installer front doors, but they are meant to land on the same experience after setup:

- Cloud server: create an Ubuntu 24.04 machine from a supported provider, pass it MOS first-boot configuration, point DNS at it, and open Suite Manager.
- Own hardware: build the MOS USB installer, boot the machine, let first boot prepare the same self-hosted setup, then open Suite Manager.

The goal is not to maintain separate VPS and homelab systems. The goal is one self-hosted MOS standard with different ways to get it onto a machine.

## Recommendation

- Start with Railway if you want the easiest experience or a quick trial.
- Choose self-hosted on a cloud server if you want independence without owning hardware.
- Choose self-hosted on your own hardware if you want the strongest local ownership.

If you are mainly curious and want the smoothest first experience, Railway is the best place to begin.
