A cloud server is a rented machine from a hosting provider.

This option is for people who want the self-hosted My Own Suite experience without buying or maintaining physical hardware.

The goal is simple: whether you install on a cloud server or on a box at home, My Own Suite should feel like the same product after setup. The cloud path will use a guided first-boot installer for a standard Ubuntu 24.04 server, then Suite Manager becomes the place where you finish setup and manage the suite.

## Who should choose this?

Choose cloud-server self-hosting if:

- you want more control over where the suite runs
- you want the option to change hosting providers later
- you want a server online without keeping hardware at home
- you are willing to follow a setup guide for choosing a provider and connecting your domain

## Tradeoffs

Compared with Railway:

- setup has more moving parts
- you own more of the server lifecycle
- the guide asks you to make a few infrastructure choices that Railway would normally hide
- updates, backups, restore, HTTPS, and customization can be managed inside your own installed suite instead of depending on a platform dashboard

In return, you get:

- more hosting flexibility
- less platform lock-in
- a clearer path to long-term independence

Compared with own hardware:

- the server is still rented infrastructure
- the provider owns the physical machine and keeps it online
- public access is usually more straightforward
- uptime and bandwidth are usually easier than a home connection

## Access style

Cloud-server self-hosting is the smoothest self-hosted option when you want to open your suite from anywhere.

The server already lives online, so connecting a domain and getting secure browser access is usually more direct than exposing a machine from a home network. Some providers may also give you a temporary HTTPS address, which can be useful while you are testing or before you connect your own domain.

This makes the cloud path a good fit when availability matters: you want the suite to feel like a normal private cloud, but you still want the apps and configuration to belong to your own installed setup.

## Privacy and ownership angle

Using a cloud server still means your data lives on hardware owned by a hosting provider, but the relationship is much closer to renting a blank machine than signing up for a consumer software ecosystem.

That matters because:

- you choose the host instead of being locked into one app company
- your suite is installed and run for you, not for the provider
- you can change providers later if pricing or terms stop fitting you
- the server provider is not the app company, dashboard, photo app, drive, calendar, or password manager
- the provider generally sees a rented server, not a built-in product account where it controls the app layer and tracks how you use each app

In other words, the machine is rented, but the suite experience is yours.

## What you will need

- A cloud server provider that supports `Ubuntu Server 24.04 LTS`
- A setup field where you can paste the MOS first-boot installer text
- A domain you control
- Access to your domain settings so the guide can point MOS addresses at the server

The intended provider requirement is intentionally generic. MOS should not need source-code changes per provider.

## Optional advanced email setup

If you already understand what an SMTP server is and have one available, the self-hosted stack can optionally reuse one shared SMTP configuration for compatible apps such as Seafile and Vaultwarden.

This is not required for the suite itself to work. It is simply an advanced extra for features like password reset emails, verification emails, and some share-link notifications.

The full technical setup belongs in the dedicated SMTP guide:

- [Optional email with SMTP](/docs/optional-email-with-smtp)

## Intended flow

The target cloud-server flow is:

1. Choose a supported provider from the MOS cloud-server list.
2. Create an `Ubuntu Server 24.04 LTS` machine.
3. Paste the MOS first-boot installer text into the provider's server setup form.
4. Point the suite domain at the server by following the DNS step in the guide.
5. Wait for first boot to install Docker, clone MOS, prepare the stack, install host agents, and start Suite Manager.
6. Open Suite Manager and finish onboarding.

This flow is not fully implemented yet. The next implementation step is a generator that creates the first-boot installer text for standard Ubuntu cloud servers.
